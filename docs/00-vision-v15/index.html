Right Model, Right Moment — Ship Faster with a Model-Agnostic Pipeline

How to get real work done while models change every week.

Core idea: use the right model for the right instance — draft locally for speed/cost, and escalate to cloud only for high-stakes steps (quality, safety, approvals).

Why now

  • Models evolve weekly: capabilities, APIs, context windows, and safety filters keep changing.
  • Budgets are tight: you need proof-of-value fast, with hard cost and latency caps.
  • Distributed teams: artifacts and handoffs matter more than meetings.
  • Regulation & risk: keep data boundaries and simple audit trails.

The problem most teams hit

You wire a promising demo into your app, then the API changes, budget spikes, or legal asks where the data went. You stall on “one perfect model” debates and delay shipping.

A simpler way

  1. Roles → agents with clear contracts (inputs/outputs you can diff and test).
  2. Routing by cost/quality/latency/context — not vendor hype.
  3. Guardrails: validators, allowlists, and bounded actions by default.
  4. Observability: logs, budgets, and diffs so you can trust changes.
  5. Escalation: local model for drafts → cloud model for sign‑off steps.

Small, concrete example

From a simple requirements YAML to sprint stories + a thin architecture note.

product: Agnostic-AI Pipeline
goals:
  - Generate stories from plain requirements
  - Keep costs under €3 per planning run
constraints:
  - No secret data leaves local machine
  - Cloud escalation only for final review

{
  "BA":     {"provider": "local", "model": "qwen2.5-coder:14b", "max_cost": 0.50},
  "ARCH":   {"provider": "local", "model": "qwen2.5:7b",        "max_cost": 0.40},
  "REVIEW": {"provider": "cloud", "model": "gpt-4o-mini",       "max_cost": 1.50, "when": "final_signoff"}
}

Outputs: planning/stories.yaml, docs/architecture.md, and logs/run.json (models, tokens, latency, cost). Escalate only when REVIEW flags ambiguity or missing acceptance tests; otherwise keep the run fully local.

How to use this at work (today)

  1. Fork the repo and run the quickstart example.
  2. Set a local provider (Ollama/LM Studio) for draft steps.
  3. Add one cloud model for final REVIEW only.
  4. Turn on budget caps and latency targets; fail fast if exceeded.
  5. Commit artifacts (stories/arch/logs) so code reviews see the reasoning.

Key takeaways

  • Right model, right moment beats “one perfect model”.
  • Artifacts > vibes: reproducible outputs make teams faster.
  • Guardrails & budgets let you ship without fear.
  • Open PoC — fork it, break it, improve it.
