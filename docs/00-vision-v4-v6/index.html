# Why an agentic, model-agnostic pipeline beats a pile of scripts

A pragmatic setup that stays flexible when your tools change

**TL;DR**

  • A lightweight, agentic, model-agnostic PoC you can try in ~30 minutes.
  • Roles as contracts (BA/Architect/Dev/QA), small guardrails, reproducible outputs.
  • Start small: run a demo, see where it helps your team, then refine it in your stack.

Most teams start with a folder of scripts. It’s fine—until the day it isn’t. The model changes, a prompt grows, a retry loop appears, and the “quick fix” eats a week. This post shows a small, honest proof-of-concept you can run today and evolve next week, without betting the farm.

Why now: the AI, economic, and cultural backdrop

  • **Models change weekly.** Capabilities, context windows, and tool APIs keep shifting. Designing **model‑agnostic** flows protects your team from churn.
  • **Budgets are tight.** You need fast **proof‑of‑value** and early **cost controls** (caps, routing, and latency targets).
  • **Distributed teams are the norm.** Clear artifacts and handoffs reduce onboarding time and unblock async work.
  • **Regulatory pressure is rising.** Keep data boundaries, simple audit trails, and reproducible outputs.

Working in a strengths/weaknesses model landscape

  • **Trade‑offs**: cost vs. quality vs. latency vs. context length vs. safety filters.
  • **IP & privacy**: understand usage terms; prefer local for drafts, escalate to cloud when justified.
  • **API volatility**: adapters and circuit breakers keep the pipeline stable while providers evolve.

What a software‑factory pipeline buys you

  • **Repeatability**: the same inputs produce inspectable artifacts (YAML, markdown).
  • **Modularity**: roles as contracts; swap an agent or model without a rewrite.
  • **Measurement**: budget/latency tracked per step; outliers are obvious.
  • **Governance**: allowlists, validators, and bounded actions by default.
  • **Community**: a shared, open PoC others can fork, improve, and reuse.

The problem

When automation starts as a pile of scripts, a few things happen over time:

1) **Script sprawl.** Every change becomes a patch. Ownership blurs and onboarding slows.

2) **Hidden lock-in.** You tailor logic to specific providers; swapping models or APIs becomes a mini-project.

3) **Inconsistent results.** Without budgets, contracts, or traces, a working run is hard to repeat.

Developers don’t need a heavy framework; they need a small, adaptable baseline that can survive model churn and keep the team moving.

A simple way forward

This PoC treats roles as small agents with contracts (clear inputs/outputs), adds budget/latency guardrails so costs don’t creep, and favors plain files + make targets so you can inspect artifacts and version them.

It’s intentionally pragmatic:

  • **Model-agnostic.** Start local (e.g., Ollama) and only escalate when needed.
  • **Reproducible.** The same prompt + inputs should yield consistent artifacts.
  • **Hackable.** It’s a starting point, not a “sealed product”. Fork it, tweak it, make it yours.

Example


# 1) Run a tiny planning flow
make ba && make plan

# 2) Inspect what changed
git status && git diff planning/

# 3) Try a guardrail (budget or latency)
export MAX_COST_USD=5
make plan

Key takeaways

  • Start with a PoC that runs in your repo today, not a platform migration.
  • Treat roles as contracts; keep agents small and observable.
  • Add budget/latency controls early; they pay back fast.
  • Make results versionable (YAML/markdown), so the team can review diffs.
  • The goal is adoption and refinement: run → learn → tweak → repeat.
