# Guardrails that matter: policies, validators, and safe actions

Enough structure to prevent the common footguns

**TL;DR**

  • Use file/action allowlists; keep them short.
  • Add preflight validators for inputs before the heavy work.
  • Bound actions to safe folders; provide one admin escape hatch.

You don’t need a fortress—just a few locks on the right doors.

The problem

Overly broad actions and unclear file permissions cause silent damage. A prompt tries to do everything, and it does it in the wrong place.

A simple way forward

  • **Allowlists.** Name the files and commands that are allowed, nothing more.
  • **Preflight.** Validate inputs early: schema, size, forbidden patterns.
  • **Bounded actions.** Confine writes to safe paths; log all changes for review.

Example


# Validate before run
make validate && make plan

Key takeaways

  • Most incidents come from overly broad capabilities; reduce scope first.
  • Preflight checks are cheap and catch 80% of mistakes.
  • Bounded writes make reviews and rollbacks straightforward.

What to do next

  • Ship a default validator and improve it with real failures from your repo.
  • Share your allowlist snippet so others can compare and adapt it.
