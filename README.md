## üöõ Loop Release Workflow
- Un ‚Äúrelease loop‚Äù corresponde a una iteraci√≥n completa desde el concepto hasta QA aprobado.
- `scripts/run_iteration.py` orquesta el flujo y asegura que cualquier estructura faltante en `project/` se restaure desde `project-defaults/`.
- `make iteration` acepta los siguientes par√°metros clave:
  - `CONCEPT="..."` ‚Üí requisito de negocio para BA/Architect.
  - `LOOPS=n` ‚Üí cu√°ntas pasadas Dev‚ÜíQA ejecutar para depurar historias pendientes.
  - `ALLOW_NO_TESTS=0/1` ‚Üí ejecuta QA en modo estricto o permisivo.
  - `ITERATION_NAME="release-X"` ‚Üí etiqueta la entrega y su snapshot.
- Cada release deja evidencia auditable:
  - `planning/` y `project/` guardados en `artifacts/iterations/<iteration>/`.
  - `summary.json` con m√©tricas de historias (`done`, `blocked`, `pending`) y configuraci√≥n usada.
- Para reintentar historias bloqueadas tras una iteraci√≥n, ejecuta `make loop MAX_LOOPS=1` con el mismo concepto; `ARCHITECT_INTERVENTION=1` (por defecto) re-planifica autom√°ticamente cuando QA falla.

### Project Defaults
- El repositorio incluye `project-defaults/`, que contiene la estructura m√≠nima para backend y frontend.
- Cada vez que un flujo llama a `common.ensure_dirs()`, cualquier archivo ausente en `project/` se replica desde este esqueleto sin sobrescribir cambios existentes.
- √ötil tras limpiar `project/` o al iniciar un nuevo concepto: garantiza que existan paquetes (`app/__init__.py`), carpetas de tests y placeholders b√°sicos.
